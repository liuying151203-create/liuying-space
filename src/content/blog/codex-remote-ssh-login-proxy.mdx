这几天我花了一些时间把 Codex 配到远程服务器上。过程里最折腾的不是安装本身，而是几个看起来很像“玄学”的问题：本地能登录，远程不行；浏览器能访问，终端不行；代理开着，但 Codex 还是连不上；VS Code Remote SSH 里重连之后，之前能用的配置又像是失效了。

这篇先记录登录和代理这一块。它更像是一份排查笔记，而不是一步到位的教程。

## 问题现象

我当时遇到的现象大概有几类：

- 在本地机器上 Codex 可以正常登录，但进入远程服务器环境后无法完成登录。
- 远程服务器可以通过某些方式访问网络，但 Codex CLI 仍然请求失败。
- VS Code Remote SSH 连接到服务器后，终端里的环境变量和本地代理设置不是一回事。
- 有些代理配置只对当前 shell 生效，重新开终端或重连后就丢了。

最容易误判的一点是：我会下意识以为“我的电脑已经开代理了，所以远程服务器也应该能用”。后来发现，这个前提本身就是错的。

## 根因：Codex 运行在哪里，网络就从哪里出去

用 VS Code Remote SSH 时，编辑器界面虽然在本地电脑上，但很多命令实际是在远程服务器上执行的。

也就是说，如果我在远程终端里运行 Codex，真正发起网络请求的是远程服务器，不是本地电脑。

这个判断也能从两个方向验证：VS Code Remote SSH 的终端会运行在远程主机上；Codex CLI 本质上也是在当前终端里启动的命令行工具。把这两件事叠在一起看，就能解释为什么“本地浏览器正常”不等于“远程 Codex 正常”。

所以本地浏览器能访问 OpenAI，不代表远程服务器里的 Codex 也能访问。远程环境至少要单独满足这些条件：

- 服务器本身能访问需要的服务地址。
- 服务器终端里有正确的代理环境变量。
- Codex 进程能继承这些环境变量。
- 登录态或配置文件在远程环境里可用。

这个认知转过来之后，问题就不再是“Codex 为什么坏了”，而是“当前这个进程到底拿到了什么网络环境”。

## 第一层检查：先确认远程终端有没有代理

我会先在远程服务器终端里看环境变量：

```bash
echo $HTTP_PROXY
echo $HTTPS_PROXY
echo $ALL_PROXY
```

有时候代理软件在本地开着，但远程 shell 里这些变量是空的。这种情况下，Codex 当然不会自动知道代理在哪里。

这一点不是 Codex 独有的问题。VS Code Remote SSH 文档里也提醒过，本地代理设置不会自动复用到远程主机；远程扩展或远程终端要使用代理，通常需要在远程环境里配置 `HTTP_PROXY` 或 `HTTPS_PROXY`。

如果远程服务器需要通过代理访问外网，可以临时设置：

```bash
export HTTP_PROXY=http://127.0.0.1:7890
export HTTPS_PROXY=http://127.0.0.1:7890
export ALL_PROXY=socks5://127.0.0.1:7890
```

这里的端口只是示例，实际要看自己的代理软件和转发方式。关键不是记住某个端口，而是确认 Codex 运行的那台机器能连到这个代理地址。

## 第二层检查：127.0.0.1 到底是谁

这个坑很常见。

在本地电脑上，`127.0.0.1:7890` 指的是本地电脑的代理端口。

但在远程服务器上，`127.0.0.1:7890` 指的是远程服务器自己的 7890 端口。除非服务器上也跑着代理，或者做了端口转发，否则它并不会自动连回本地电脑。

如果使用 SSH 端口转发，可以把本地代理端口转到远程环境，例如：

```bash
ssh -R 7890:127.0.0.1:7890 user@server
```

这样远程服务器访问自己的 `127.0.0.1:7890` 时，才可能被转发回本地电脑的代理端口。

这也是我后来理解代理问题的关键：不是“有没有代理”，而是“当前进程看到的代理地址是否真的可达”。

## 第三层检查：登录态不要随便复制

登录问题还有一部分和 Codex 的本地配置、认证状态有关。

如果把本地的配置目录复制到远程服务器，要特别小心：里面可能包含和账号、会话、授权相关的敏感信息。它不应该提交到 Git，也不应该放到多人共用的服务器目录里。

更稳妥的做法是：

- 优先在远程环境里按正常流程登录。
- 确认配置文件权限只对当前用户可读写。
- 不把认证文件写进项目目录。
- 不在博客或公开笔记里贴 token、cookie、完整配置文件。

我的经验是，登录问题不要一上来就靠复制文件解决。先确认网络可达，再确认 Codex 能正常打开登录流程，最后再处理持久化配置。

## 我现在的排查顺序

后面再遇到类似问题，我会按这个顺序看：

1. 当前 Codex 是在本地运行，还是在远程服务器运行？
2. 运行 Codex 的那个终端里，代理环境变量有没有值？
3. 代理地址对当前机器是否真的可达？
4. VS Code Remote SSH 重连后，这些变量是否还存在？
5. 登录态是否在正确的用户目录下？
6. 配置文件权限是否安全？

这个顺序的好处是，它把问题从“工具不好用”拆回了几个可以验证的小问题。
